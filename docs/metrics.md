# Continuum Observer Metrics

Continuum measures and reports on various dimensions of coding agent performance, efficiency, and safety.

## 1. Context and Telemetry

- **Task Input**: The initial prompt and any additional arguments.
- **Commands Run**: Shell commands executed by the agent (e.g. `npm test`, `git status`).
- **Files Inspected**: Files read or queried during the run.
- **Files Modified**: Files changed, created, or deleted by the agent.
- **Tool Invocations**: The specific tools (functions) invoked by the agent.

## 2. File Change Attribution Confidence

Continuum categorizes file modifications with an **Attribution Confidence**:

- **HIGH**: The file was completely clean (unmodified and tracked) before the run, and the agent modified it. We are certain the agent made these changes.
- **MEDIUM**: The file is newly untracked (created during the run). We are fairly certain the agent created it, though the developer could have theoretically created it simultaneously.
- **LOW**: The file was already dirty (modified) in the working tree before the agent ran. The agent made further modifications. We cannot reliably isolate the agent's specific diff from the developer's pre-existing uncommitted changes.
- **UNKNOWN**: Status could not be computed (e.g. Git failure).

## 3. Resource Usage (Cost)

When the agent provides this information via telemetry (e.g. Gemini CLI's `usage` events):

- **Input Tokens**: Number of tokens in the prompt, context, and file contents.
- **Output Tokens**: Number of tokens generated by the model.
- **Total Tokens**: Sum of input and output tokens.
- **Duration**: The elapsed wall-clock time from task start to task completion.

## 4. Empirical Efficacy

- **Validation Test Outcomes**: Continuum runs a user-defined test command before and after the agent run.
  - **Fixed Tests**: Tests that were failing before the run, but pass after.
  - **Broken Tests**: Tests that were passing before the run, but fail after.
  - **Unchanged**: Tests that maintained their prior state (passing->passing, failing->failing).

## 5. Subjective Developer Efficacy

Captured via the `continuum outcome` command:

- **Accepted**: The developer accepted the agent's work as-is, without modifications.
- **Modified**: The developer accepted the core of the agent's work, but had to make manual corrections or adjustments before committing.
- **Rejected**: The agent's work was incorrect, unhelpful, or harmful, and was discarded by the developer (e.g. via `git reset --hard`).
